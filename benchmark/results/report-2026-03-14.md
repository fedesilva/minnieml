# MML benchmark report
Date: 2026-03-14

Previous report: [2026-02-07](report-2026-02-07.md)

## Changes since last report

Compiler changes that could affect codegen:

- **TCO fix for nested conditional branches** (d4c51f7, f908ab9): Tail call optimization now correctly handles recursive calls that appear inside nested `if`/`else` branches when there are multiple call sites. The previous pass would fail to convert them to jumps in certain patterns.
- **Global borrow-only ownership semantics** (d4c51f7): Compile-time enforcement of ownership rules — no codegen impact expected.

## Preliminary notes

### Unsafe array access

Most benchmarks use `unsafe_ar_int_*` intrinsics for array access: direct LLVM load/store with no bounds checking. This puts MML on equal footing with C. Rust and Go check bounds by default (Go has BCE, which hasn't helped much here — see matmul).

The eventual plan is for the type system to verify bounds at compile time using effects tracking and affine types. For now these numbers are a baseline.

### Data types

Benchmarks use `int64` throughout, since the specialised `IntArray` type only fully supports `int64`. The generalising typechecker is still in progress.

## Sieve of Eratosthenes

Finds all primes up to 1,000,000.

- `sieve-mml`: tail recursion, `unsafe_ar_int_*` array access
- `sieve-c`: `malloc` + `while` loops, branchless count
- `sieve-rs`: idiomatic Rust, `Vec<i64>`, bounds checking on
- `sieve-go`: idiomatic Go, slices

|:---|---:|---:|---:|---:|
| `bin/sieve-mml` | 5.6 ± 0.6 | 4.7 | 7.2 | 1.00 |
| `bin/sieve-c` | 6.4 ± 1.5 | 4.7 | 10.3 | 1.14 ± 0.29 |
| `bin/sieve-rs` | 6.7 ± 0.7 | 5.6 | 8.2 | 1.20 ± 0.18 |
| `bin/sieve-go` | 8.9 ± 0.8 | 7.6 | 11.0 | 1.59 ± 0.22 |

MML comes out fastest this run. The C result has large variance (±1.5ms vs ±0.6ms for MML), so the 14% gap is mostly noise — the min times are identical at 4.7ms. Read this as another tie, not a win. Go trails by ~59%, consistent with previous runs.

## N-Queens

N=12, counting all 14,200 solutions. Backtracking with array board state and TCO in the inner `is_safe` loop.

| Command | Mean [ms] | Min [ms] | Max [ms] | Relative |
|:---|---:|---:|---:|---:|
| `bin/nqueens-c` | 86.3 ± 6.1 | 79.7 | 103.8 | 1.00 |
| `bin/nqueens-mml` | 105.8 ± 4.2 | 99.7 | 118.8 | 1.23 ± 0.10 |
| `bin/nqueens-go` | 137.4 ± 4.1 | 130.8 | 146.3 | 1.59 ± 0.12 |

MML improved from 1.57x C to 1.23x C — the MML runtime dropped from 144ms to 106ms while C moved from 92ms to 86ms. The cause is almost certainly the TCO fix for nested conditional branches. The N-Queens `solve` function has exactly this pattern: recursive calls inside nested `if`/`else` with multiple call sites. Previously those weren't converted to jumps; now they are. MML also now clearly beats Go (106ms vs 137ms) rather than barely edging it out (144ms vs 146ms in February).

## Summary

### Wins

- **N-Queens: 1.57x → 1.23x C.** MML went from 144ms to 106ms. The TCO fix for nested conditional branches with multiple call sites is the likely cause — the `solve` function has exactly that pattern. MML now beats Go on this benchmark (106ms vs 137ms).

### Stable

- **Optimised matmul at C parity** (43.1ms each). Holding since the `noalias` fix in February.
- **Sieve**: another tie with C, consistent across all three reports.
- **Euclidean**: ~1.09x C, within noise of parity.

### Regressions persisting

- **Ackermann: 1.21x C.** The ~22% gap introduced by attribute group infrastructure in February is unchanged. Still the most obvious open issue for call-heavy code with no arrays.
- **Naive matmul O2/O3**: still running at ~250ms instead of ~100ms due to the noalias-triggered loop transform regression. The optimised (i-k-j) variant benefits from noalias; the naive strided variant does not.

### To investigate

1. **Ackermann**: Diff the LLVM IR before and after attribute groups. If adding `#0` to every function is the cause, the fix is straightforward: only emit attribute groups for functions that actually use non-default attributes.
2. **Naive matmul O2/O3**: The regression is noalias interacting with LLVM's loop optimiser. Compare the IR at O1 (where it's fast) and O2 (where it's slow) to find which transform is responsible.
3. **N-Queens gap**: 1.23x is better but C is still 23% faster. Worth profiling now that the obvious TCO issue is fixed, to see whether the remaining gap is call overhead, array access patterns, or something else.
